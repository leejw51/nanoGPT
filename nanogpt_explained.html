nanoGPT, explained from scratch

A ~600-line repo that trains and runs a GPT. Two files do the heavy lifting: model.py defines the network, train.py optimizes it. This page walks the whole path — install → data → train → sample — explains every tensor shape, and shows exactly what to edit to print shapes at each step.

PyTorchGPT-2 architecture~10.6M–1.5B params char or BPE tokenssingle file model

1 · Overview

nanoGPT is a decoder-only Transformer (the GPT-2 architecture) you can read in one sitting. The job of the model is dead simple: given a sequence of tokens, predict the next token. Do that well, and you can generate text by sampling one token at a time and feeding it back in.

1. Prepare data

data/*/prepare.py

Turn raw text into a stream of integer token ids saved as train.bin / val.bin.

2. Train

train.py + model.py

Repeatedly: grab a batch, predict next tokens, measure error (loss), nudge the weights.

3. Sample

sample.py

Load the trained weights and autoregressively generate new text.

Mental model: the entire network is a function idx → logits. Input idx is a grid of token ids of shape [batch, time]; output logits is a probability score for every vocabulary word at every position, shape [batch, time, vocab]. Everything else is plumbing around that one function.

Heads up: the README marks this repo as old/deprecated in favor of nanochat. nanoGPT is still the cleanest place to learn how a GPT works, which is what this page is for.

4 · Architecture

The model is a stack of identical Transformer blocks wrapped between embeddings (in) and a linear head (out). Here is the whole thing, top-down.

The config

Every shape in the network is derived from these six numbers:

model.py
FieldSymbolWhat it controls
block_sizeTmax context length (how many past tokens it can see)
vocab_sizeVnumber of distinct tokens
n_embdCembedding width — the "thickness" of every vector
n_headnhattention heads (must divide C; head size = C/nh)
n_layerLhow many blocks are stacked
dropout, bias—regularization / whether Linears & LayerNorms have bias

Top-level assembly (the GPT module)

transformer = ModuleDict(...)
wte token embedding  Embedding(V, C)
id → vector
wpe position embedding  Embedding(T, C)
slot → vector
drop dropout
h = ModuleList([ Block × n_layer ])
each Block: ln_1 → attn → (+) → ln_2 → mlp → (+)
ln_f final LayerNorm
lm_head  Linear(C, V, bias=False)  → logits
vector → vocab scores
Weight tying: wte.weight = lm_head.weight. The same matrix maps ids→vectors on the way in and vectors→scores on the way out. Saves V×C params and trains better.

Component 1 — CausalSelfAttention

This is where tokens "look at" earlier tokens. Causal = a token may only attend to itself and the past, never the future (otherwise it would cheat at next-token prediction).

model.py
  • One matmul makes Q, K, V. c_attn projects each token vector (C) to 3C, then splits into query/key/value.
  • Reshape into heads. C is sliced into nh heads of size hs = C/nh; shape becomes [B, nh, T, hs] so each head attends independently.
  • Attention scores. q @ kᵀ gives a [B, nh, T, T] matrix — how much each token attends to each other token. Scaled by 1/√hs, causally masked, softmaxed.
  • Weighted sum of values. att @ v → [B, nh, T, hs], then heads are concatenated back to [B, T, C] and projected by c_proj.
  • Flash attention (scaled_dot_product_attention, PyTorch ≥ 2.0) fuses score+mask+softmax+matmul into one fast kernel — same math, no explicit [B,nh,T,T] tensor in memory.

Component 2 — MLP

A per-token 2-layer feed-forward net. Expand to 4×, apply GELU non-linearity, project back. This is where most of the model's "knowledge" capacity lives.

model.py

Component 3 — Block (the repeated unit)

A block = attention + MLP, each with a pre-LayerNorm and a residual (skip) connection. The x + is the residual: the block learns a delta to add to the running stream, which makes deep stacks trainable.

model.py

The full forward pass

model.py
Train vs. inference branch: if targets are given (training), it computes logits for all positions and a cross-entropy loss. If not (generation), it only runs lm_head on the last position — that's all you need to predict the next token, and it's much cheaper.

7 · Training loop

train.py is one big while True loop. Strip away DDP/logging and it's just this:

train.py · simplified

The data loader

No fancy DataLoader — just random windows into a memory-mapped array of token ids. x is a window, y is the same window shifted right by one (the "next token" target).

train.py
Why y is x shifted by 1: at position t the model sees x[0..t] and must predict x[t+1], which is exactly y[t]. One forward pass produces a prediction at every position at once — that's why training is efficient.

Key mechanics

MechanismWhy
Gradient accumulation (gradient_accumulation_steps) Run several micro-batches, summing gradients before one optimizer step → simulate a huge batch on small VRAM.
Mixed precision (autocast + GradScaler) Compute in bf16/fp16 for speed; the scaler prevents fp16 gradient underflow.
Cosine LR schedule with warmup (get_lr) Ramp learning rate up over warmup_iters, then cosine-decay to min_lr.
Gradient clipping (grad_clip) Cap gradient norm at 1.0 so a bad batch can't blow up the weights.
AdamW + selective weight decay 2D tensors (matmuls, embeddings) get weight decay; biases & LayerNorms don't.
Eval & checkpoint every eval_interval Estimate train/val loss over many batches; save ckpt.pt when val loss improves.
DDP (torchrun) One process per GPU; gradients are all-reduced (averaged) across GPUs each step.
one optimization step
set learning rate get_lr(iter)
× grad_accum: logits,loss = model(X,Y); loss.backward(); prefetch next batch
clip grads → optimizer.step() → zero_grad()
every eval_interval: estimate_loss(); save ckpt

8 · Inference / sampling

Generation is a loop around the forward pass. Each step: predict next-token logits, sample one token, append it, repeat. This is autoregressive decoding.

model.py
  1. Crop context to the last block_size tokens (the model can't see further back).
  2. Forward → take logits at the final position only: logits[:, -1, :], shape [B, V].
  3. Temperature divides the logits: <1 sharpens (safer), >1 flattens (wilder).
  4. top-k zeroes out all but the k most likely tokens before sampling.
  5. Softmax → multinomial sample → one new token id, appended to the sequence.
@torch.no_grad() on generate() disables gradient tracking — inference doesn't learn, so skipping the backward bookkeeping saves memory and time.

sample.py wraps this: it loads the checkpoint, rebuilds the encoder/decoder from meta.pkl (char-level) or falls back to GPT-2 BPE via tiktoken, encodes your start prompt, calls generate(), and decodes the ids back to text.
